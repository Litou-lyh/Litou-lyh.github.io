1. Summary

1.1 What is Hadoop HDFS

The Hadoop Distributed File System (HDFS) is designed to store very large data

sets reliably, and to stream those data sets at high bandwidth to user applications.

1.2 Architecture

1.2.1 NameNode

  • Stores metadata of files
  • Files and directories are represented on the NameNode by inodes.
  • One NameNode for each cluster

1.2.2 Image and Journal

  • The inodes and the list of blocks that define the metadata of the name system are

called the image.

  • NameNode keeps the entire namespace image in RAM
  • persistent record of the image stored in the NameNode’s local native file system is called a checkpoint
  • The NameNode records changes to HDFS in a write-ahead log called the journal in its local native filesystem. Batched flush-and-sync procedure is applied to update metadata.
  • Checkpoint files will never be modified
  • New Checkpoint files can be created when startup the NameNode
  • redundant copies of the checkpoint and journal are typically stored on multiple independent local volumes and at remote NFS servers.

1.2.3 DataNode

  • Connection to NameNode
  • A namespace ID is assigned to each node in the same cluster. It protects the integrity of the cluster (Decline connection if namespace IDs do not match)
  • DataNode registers with NameNode and is assigned with a permanent Storage ID.
  • During normal operation DataNodes send heartbeats to the NameNode

1.2.4 HDFS Client

  • User applications access the filesystem using the HDFS client, a library that exports the HDFS filesystem interface
  • When an application reads a file, the HDFS client first asks the NameNode for the list of DataNodes that host replicas of the blocks of the file
  • The client organizes a pipeline from node-to-node and sends the data. When the first block is filled

2. Prof’s Questions

a. How does HDFS know when a failure occurs? When a DataNode fails, how does HDFS recover data?

DataNodes send heartbeats to NameNode periodically. When a failure occurs in the dataNode, NameNode does not receive heartbeats and thus detects its failure. The NameNode creates a new replica of those blocks on other DataNodes.

b. Why is a replication factor less than three not recommended in HDFS?

Replication factors are set to be greater than or equal to 3 for higher consistency. If the replication factor is less than 3, when one of the replicas corrupts, the system can not tell which of the replicas stores the correct data. But if the factor is greater than or equal to 3, the system can detect and correct single corruption by adopting the majority and correcting the corrupted one.

c. Does HDFS opt for Consistency or Availability (based on the CAP theorem)? Briefly explain your rationale.

HDFS opts for consistency. HDFS ensures consistency by the design of NameNode and multiple DataNode. Then Availability cannot be satisfied since HDFS chooses C & P. When some DataNodes cannot be reached or the NameNode fails, the data may become unavailable or the system cannot respond in time.
